## 4. 今日理论知识

### 4.1 FastAPI Minimal Service

#### 它是什么

FastAPI Minimal Service 是把原来只能通过 CLI 或脚本调用的问诊 workflow，封装成 HTTP API。用户或前端可以通过接口创建 session、提交一轮问诊、查询状态和报告。

#### 为什么本项目需要它

中医问诊辅助系统不能一直停留在 `python scripts/run_graph_demo.py` 这种脚本形态。API 层让系统具备后端服务雏形，后续才能对接前端、测试、演示或其他系统。

#### 它在项目中的位置

API 层。

#### 对应代码

- `app/api/main.py`
- `GET /health`
- `POST /sessions`
- `POST /sessions/{session_id}/turn`
- `GET /sessions/{session_id}/state`
- `GET /sessions/{session_id}/report`

#### 它起到的作用

没有 API 层，系统虽然有 LangGraph workflow，但外部无法以稳定接口调用，也无法形成“服务化项目”的工程表达。

#### 实现原理

`app/api/main.py` 创建 FastAPI app。`POST /sessions` 调用 session runtime 创建 `ApiSession`，`POST /sessions/{id}/turn` 读取 session 后调用 `run_consultation_graph()`，再组装 `TurnResponse` 返回。

#### 面试怎么讲

我在 P1.1 没有重写问诊逻辑，而是用 FastAPI 给现有 LangGraph workflow 包了一层最小服务壳。API 只负责请求校验、session 编排和响应组装，核心抽取、风险规则、RAG 和安全后检查仍复用 P0 已验证流程。

### 4.2 Pydantic API Schema

#### 它是什么

Pydantic API Schema 是接口请求体和响应体的结构化契约，用类型和字段约束保证输入输出稳定。

#### 为什么本项目需要它

医疗辅助场景不能让任意字符串随便进入 workflow。比如 `extractor_mode` 只能是 `real_llm / fake / fallback`，空 `user_input` 不能进入问诊流程。

#### 它在项目中的位置

API Schema 层。

#### 对应代码

- `app/api/models.py`
- `CreateSessionRequest`
- `TurnRequest`
- `TurnResponse`
- `SessionStateResponse`
- `SessionReportResponse`

#### 它起到的作用

它让 API response 有稳定字段，例如 `metadata.graph_runtime`、`metadata.extractor_mode`、`metadata.fallback_used`、`safety_disclaimer`。这对前端、测试和面试表达都很重要。

#### 实现原理

FastAPI 自动根据 Pydantic model 校验请求和序列化响应。不合法 `extractor_mode` 会返回 422；空输入会被 API 层拒绝。

#### 面试怎么讲

我用 Pydantic 固化 API 契约，保证问诊接口的输入和输出都可预测。这样既能减少脏数据进入 LangGraph，也让前端和测试可以依赖稳定 JSON 结构。

### 4.3 Session Runtime

#### 它是什么

Session Runtime 是 API 层的运行态会话管理模块，负责保存 `session_id`、`extractor_mode`、`rag_enabled`、`turn_count`、当前 `RunState`、turn 历史和最终报告。

#### 为什么本项目需要它

问诊是多轮流程。用户第一轮说“胃胀”，第二轮说“一周了”，系统必须记住上一轮状态，否则会话无法连续。

#### 它在项目中的位置

API Runtime 层。

#### 对应代码

- `app/api/session_runtime.py`
- `ApiSession`
- `create_session()`
- `get_session()`
- `append_turn()`

#### 它起到的作用

它把 HTTP 的无状态请求，连接到多轮问诊的有状态流程。没有它，API 每次调用都会像第一次问诊。

#### 实现原理

P1.1 阶段用内存字典和 `RLock` 管理 session。P1.2 后，`get_session()` 优先从 SQLite 恢复 session/state，再写回内存缓存。

#### 面试怎么讲

HTTP 本身是无状态的，但问诊必须有上下文。我设计了 session runtime 管理 `RunState`，让每次 turn 都能基于上一轮状态继续执行，同时在 P1.2 接入 SQLite 支持重启恢复。

### 4.4 SQLite Persistence

#### 它是什么

SQLite Persistence 是用本地 SQLite 文件保存 API session 的最小持久化能力。

#### 为什么本项目需要它

P1.1 的 in-memory session 服务一重启就丢失状态。P1.2 解决的是“服务重启后还能恢复会话”的最小工程问题。

#### 它在项目中的位置

Persistence 层。

#### 对应代码

- `app/api/sqlite_store.py`
- `initialize_database()`
- `fetch_session()`
- `append_turn_and_update_state()`
- `TCM_API_DB_PATH`

#### 它起到的作用

保存三类数据：

- `sessions`：session 元信息
- `session_states`：当前完整 `RunState` JSON
- `turns`：每轮输入和响应摘要

#### 实现原理

使用 Python 标准库 `sqlite3`。每次提交 turn 后，在同一个事务里插入 turn、更新当前 state、更新 session 的 `updated_at`，避免“turn 写了但 state 没更新”的半成功状态。

#### 面试怎么讲

P1.2 我没有直接上复杂 ORM，而是用标准库 SQLite 做最小持久化。三张表分别存 session 元数据、当前状态 JSON 和 turn 历史，提交一轮问诊时用事务保证 turn 与 state 一致，满足本地 demo 和工程验收需要。

### 4.5 LangGraph Workflow 复用

#### 它是什么

LangGraph Workflow 是 P0 已经验证过的问诊主流程，包括输入规范化、单轮抽取、状态合并、风险规则、追问决策、RAG、报告生成和安全后检查。

#### 为什么本项目需要它

P1.1/P1.2 的目标是服务化和持久化，不是重写问诊逻辑。复用 LangGraph 可以保证 P0 能力不回退。

#### 它在项目中的位置

Workflow / Graph 层。

#### 对应代码

- `app/graphs/consultation_graph.py`
- `run_consultation_graph()`
- `NODE_SEQUENCE`
- `app/graphs/consultation_nodes.py`

#### 它起到的作用

API 层每次收到用户输入后，仍然走同一条 graph 主流程，而不是在 API 里复制一套判断逻辑。

#### 实现原理

`POST /sessions/{id}/turn` 从 session 中取出当前 `RunState`，调用 `run_consultation_graph()`，再把 graph 输出写回 session 和 SQLite。

#### 面试怎么讲

我把 API 层设计成薄封装：它不懂问诊业务细节，只调用已有 LangGraph workflow。这样可以保持 P0 评估闭环不被破坏，也避免服务化阶段引入两套业务逻辑。

### 4.6 Secret Redaction 与医疗安全边界

#### 它是什么

Secret redaction 是防止 API key、token、`.env` 内容进入响应、日志或数据库；医疗安全边界是防止系统输出诊断、处方、治疗方案。

#### 为什么本项目需要它

项目既涉及 LLM API key，也涉及医疗辅助场景。安全边界如果不明确，项目很容易从“问诊辅助”被误讲成“诊断系统”。

#### 它在项目中的位置

API 层 / Persistence 层 / Safety 层。

#### 对应代码

- `app/api/main.py`
- `app/api/sqlite_store.py`
- `app/safety/report_safety.py`
- `docs/P0_FINAL_BASELINE.md`
- `docs/P1_2_SQLITE_PERSISTENCE_REPORT.md`

#### 它起到的作用

保证 API 不返回 `OPENAI_API_KEY`、`sk-`、`.env` 内容；保证响应包含安全免责声明；保证 FinalReport 不输出诊断、处方或治疗方案。

#### 实现原理

API 响应统一带 `safety_disclaimer`，SQLite 持久化前会对常见 secret-like pattern 做脱敏，报告层仍经过 `report_safety.py` 后检查。

#### 面试怎么讲

我在服务化和持久化阶段都保留了安全边界：接口不泄露密钥，数据库不保存 secret，最终报告不输出诊断和处方。项目定位始终是问诊信息整理和风险提示。

## 11. 面试表达沉淀

### 30 秒版本

TCM-Assistant 是一个中医问诊辅助系统，不是诊断系统。P1.1 我把已有 LangGraph 问诊 workflow 封装成 FastAPI 最小服务，提供创建 session、提交 turn、查询 state/report 的 API。P1.2 我用 SQLite 做最小持久化，保存 session 元信息、当前 `RunState` JSON 和 turn 历史，让服务重启后仍能恢复会话状态，同时不保存 secret、不新增 Memory 或诊断能力。

### 1 分钟版本

这个阶段的重点是工程服务化和状态持久化。API 层用 FastAPI 和 Pydantic 定义稳定接口，`POST /sessions` 创建问诊会话，`POST /sessions/{id}/turn` 提交一轮症状输入，但业务逻辑仍复用 P0 已验证的 `run_consultation_graph()`。P1.1 先用 in-memory session，P1.2 再新增 SQLite，三张表分别保存 session 元数据、最新 state JSON 和 turn 历史。每次 turn 在一个事务里同时写入 turn 和更新 state，保证恢复时状态一致。整个阶段没有引入 Storage 大系统、Memory、Embedding、多 Agent 或 Web UI。

### 3 分钟版本

P0 阶段已经完成结构化抽取、LangGraph 工作流、风险规则、Hybrid RAG、安全后检查和评估闭环。P1.1 的目标不是改问诊算法，而是把这条 workflow 服务化。我新增 FastAPI 最小服务，接口包括 `/health`、`/sessions`、`/sessions/{id}/turn`、`/state` 和 `/report`。请求和响应用 Pydantic models 固定结构，防止非法 extractor mode 和空输入进入 workflow。

P1.2 解决的是服务重启后会话丢失的问题。我用标准库 `sqlite3`，没有引入 ORM。SQLite schema 保持最小：`sessions` 保存元信息，`session_states` 保存当前完整 `RunState` JSON，`turns` 保存每轮输入和响应摘要。提交一轮输入时，API 先恢复 session，再调用 LangGraph，然后在一个事务里写入 turn 并更新 state。这样满足本地 demo 和工程验收，同时不把它包装成长期 Memory。

安全方面，API response 带安全免责声明，报告仍经过 safety 后检查；SQLite 持久化前对 secret-like 内容脱敏，不保存 API key、token 或 `.env` 内容。验收上，P1.1 API 测试、P1.2 持久化测试、P0 回归、real LLM eval 和 SFT eval 都有报告记录。

### 简历表达
